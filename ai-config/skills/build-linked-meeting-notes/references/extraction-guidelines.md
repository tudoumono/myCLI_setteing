# 抽出・要約ガイドライン

## 目次

- 1. トピック分割の基準

- 3. 決定事項とアクションの抽出

- 6. 出典の残し方

## 1. トピック分割の基準

次のいずれかが起きたら、新しいトピック候補として扱う。

- 議題が切り替わった
- 話者が「次に」「別件」「続いて」などで転換した
- 決定フェーズから実装詳細フェーズへ移った
- 同じ議題でも、論点（費用/日程/担当）が大きく変わった

分割しすぎを避けるため、5分未満の細かな脱線は直前トピックへ吸収してよい。

## 2. 要約の書き方

### トップページ要約

- 3〜8行を目安にする。
- 会議全体の目的、主要な結論、未解決事項を含める。
- 詳細な経緯は書きすぎない（トピック詳細へ誘導する）。

### トピック要約

- 2〜4行を目安にする。
- 「何を議論したか」と「結果（決定/保留）」を入れる。
- 具体名（機能名、案件名、担当者）が重要なら残す。

## 3. 決定事項とアクションの抽出

### 決定事項（Decision）

次に該当する発言を抽出する。

- 採用・却下・延期などの意思決定
- 仕様確定
- スケジュール確定
- 担当確定（ただし作業依頼の詳細はアクションにも書く）

### アクションアイテム（Action）

次の項目をできるだけ埋める。

- `Axx` ID
- 内容
- 実施者（担当者）
- 期限（不明なら `[要確認]`）
- 関連トピックID

実施者（担当者）の扱い:

- 会議中に「誰がやるか」が整理されている場合は、その名前やロールを記載する。
- トランスクリプト上で実施者が不明な場合は、空欄にせず `不明` と記載する。
- 推測で人名を入れない。候補があるだけなら `不明` のままにして補足で書く。

1つのアクションが複数トピックにまたがる場合は、主トピックを1つ決めて他トピック側から参照する。

## 4. メモとの紐づけ方

ユーザーが別途メモ（箇条書き、手書き起こし、Slackメモ等）を渡した場合に適用する。

1. メモを項目単位に分解する。
2. `M01`, `M02` のようにIDを振る。
3. 各メモ項目を最も関係の深いトピックへ紐づける。
4. 根拠が弱い場合は複数候補を列挙し、`[要確認]` を付ける。

優先順位:

- 明確なキーワード一致
- 話者名一致
- タイムスタンプ近接
- 文脈一致

## 5. 曖昧さの扱い

以下のケースは推測で補完しすぎない。

- 人名/固有名詞が聞き取れない
- 金額/日付/数量が不明瞭
- 決定に見えるが合意が取れていない
- 「誰がやるか」が曖昧

書き方の例:

- `実施者: 不明`
- `期限: [要確認]（3月上旬と言及あり）`
- `決定状況: 保留（合意表現が不明確）`

## 6. 出典の残し方

再確認しやすいように、トピック詳細の末尾に出典を残す。

推奨形式:

```md
- 00:14:32 佐藤: 「来週金曜までにベータ公開したい」
- 00:16:05 鈴木: 「QAの工数次第なので、日程は一旦仮置き」
```

タイムスタンプがない場合は以下のいずれかで代替する。

- 発話順番号
- 段落番号
- メモID（`Mxx`）

重要な数値や期限は、要約だけでなく出典にも残す。
